## ROOT CAUSES

### 1. **Malware Infection**
- Development machine compromised
- Unclear vector of infection
- No endpoint protection or detection in place

### 2. **Scattered API Keys**
- Supabase credentials hardcoded in **11 different HTML/JS files**
- No centralized configuration management
- Rotating a single key required finding and updating ~dozen files
- Keys embedded in client-side code (security anti-pattern)

### 3. **No Health Monitoring**
- No automated service health checks
- No alerting when services went down
- First detection was manual observation after ~6 hours
- No telemetry on which services were affected

### 4. **Supabase API Key Format Migration**
- Legacy `anon` key format deprecated by Supabase
- New `sb_publishable_*` format required
- Migration was incomplete across codebase
- Multiple attempts to switch formats caused cascading failures

### 5. **No Runbook or Incident Procedures**
- No documented key rotation process
- No single source of truth for "where keys live"
- Recovery required searching git history and guessing

---

## WHAT WENT WRONG

1. **Attack surface too large:** 11 files with hardcoded credentials = 11 places for malware to steal keys
2. **No observability:** System was down for 6+ hours before detection
3. **Slow recovery:** No runbook meant every step had to be figured out during the outage
4. **No defense in depth:** Single compromise took down the entire system
5. **Manual key rotation:** Required editing dozens of files, high error rate, slow

---

## FIXES APPLIED (Resilience Sprint)

### 1. **Centralized Configuration**
- Created `/api/config.js` endpoint serving `SUPABASE_URL` and `SUPABASE_PUBLISHABLE_KEY` from Vercel env vars
- Refactored **all 11 client-side files** to fetch config from `/api/config` on page load
- **Result:** Rotating a key now requires updating ONE place (Vercel dashboard) instead of 11 files

**Files refactored:**
- `signin.html`
- `forgot-password.html`
- `set-password.html`
- `founding.html`
- `calculator.html`
- `agents/index.html`
- `coordinators/index.html`
- `sales-modal.js`
- `assets/calculator-widget.js`

**Pattern:**
```javascript
const { supabaseUrl, supabaseKey } = await fetch('/api/config').then(r => r.json())
const supabase = createClient(supabaseUrl, supabaseKey, { auth: { persistSession: true } })
```

### 2. **Health Monitoring**
- Created `/api/health.js` — checks Supabase, Telegram, ElevenLabs, Creatomate every request
- Returns HTTP 200 if all services `ok`, HTTP 503 if any degraded
- Response includes per-service status, response time, timestamp

**Example healthy response:**
```json
{
  "status": "ok",
  "services": {
    "supabase": "ok",
    "telegram": "ok",
    "elevenlabs": "ok",
    "creatomate": "ok"
  },
  "responseMs": 487,
  "timestamp": "2026-05-08T08:20:00.000Z"
}
```

### 3. **Automated Alerting**
- Created `/api/alert-health.js` — calls `/api/health`, sends Telegram alert if any service is down
- Configured Vercel cron to run every 5 minutes: `*/5 * * * *`
- Alert message format: `🚨 DOSSIE ALERT\n❌ supabase: error:401\n2026-05-08T...`
- **Result:** Heath gets notified within 5 minutes of any outage

### 4. **RUNBOOK.md**
- Documented every API key name, where it lives, how to rotate it
- Standard rotation checklist: delete → re-add → pull → redeploy → verify
- Emergency procedures for each failure mode
- Health monitoring reference
- **Result:** Any future key rotation is a copy-paste operation, not archaeology

### 5. **Startup Scripts**
- Created `start-claudy.ps1` and `start-claudy.bat` for one-click Claude Code restart
- Eliminates manual command entry after token rotation
- **Result:** Recovery time reduced from ~10 minutes to ~10 seconds

### 6. **INCIDENT-2026-05-08.md** (This Document)
- Timeline of events
- Root causes documented
- Every fix catalogued
- Lessons learned preserved
- **Result:** Future incidents can reference this playbook

## LESSONS LEARNED

### 1. **Centralize secrets from day one**
- Hardcoded keys are a security liability AND an operational nightmare
- Every duplicated secret is a future rotation headache
- **Action:** Never hardcode credentials again. Use env var → API pattern.

### 2. **Observability is not optional**
- A system without health checks is a system you can't trust
- Detection lag = customer impact duration
- **Action:** Every new service gets a health check probe on day one.

### 3. **Runbooks save hours**
- 10 minutes writing a runbook saves 6 hours during an outage
- Memory fails under pressure; checklists don't
- **Action:** Document key operations before they're needed.

### 4. **Automate alerts**
- "I'll check it manually" is a lie we tell ourselves
- Humans don't scale; cron jobs do
- **Action:** If it matters, alert on it.

### 5. **Test failure modes**
- We didn't know alerting worked until the system broke
- **Action:** Monthly drill: rotate one key, verify health check catches it.

### 6. **Defense in depth**
- Single point of failure = single point of catastrophe
- **Action:** Next sprint: add API key rotation automation, backup alert channels.

## ROOT CAUSES
1. Upload limit set to default (too low) — never tested with real files
2. Opus model ID hardcoded as 'claude-opus-4-7' (invalid) — large PDF scanning broken before launch
3. Media/*.mp4 files committed to repo — caused 250MB Vercel bundle failures
4. No pre-launch testing with real contract PDFs

## FIXES APPLIED
1. Frontend + API upload limit raised to 25MB
2. Opus model ID corrected to claude-opus-4-5-20251101
3. Media/ added to .vercelignore
4. Large PDFs (>5MB) now routed to Opus with max_tokens 8192

## LESSONS LEARNED
1. Never commit binary/media files to repo — use Supabase Storage or CDN
2. Always validate model ID strings against Anthropic docs before deploying
3. Test with real customer file sizes before onboarding paying customers
4. Monitor Vercel deploy success emails — failures were silent for days
5. Add .vercelignore to project setup checklist from day one

## PREVENTION RULES (Claude Code must follow)
- NEVER commit video, audio, or image files >1MB to repo
- ALWAYS verify Anthropic model strings before using in production
- ALWAYS add Media/, videos/, assets/video/ to .vercelignore at project start
- Run a pre-launch checklist before any new customer onboards
